# DS Audit Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Produce the DS audit diagnostic (`docs/ds-audit-2026-04-20.md`) and fix backlog (`docs/ds-audit-fix-backlog.md`) per the approved spec. Read-only — no code changes, no UI changes.

**Architecture:** Hybrid methodology. A single Explore subagent runs mechanical counts (Pass 1). The main session then does pattern extraction from top pages (Pass 2), a consolidation scan of all DS components (Pass 3), and synthesis into the four report sections + fix backlog (Pass 4). All intermediate artefacts live in `docs/ds-audit/` and are committed so the scan is re-runnable.

**Tech Stack:** No code stack — this is a markdown-producing research task. Tools used: Bash (`grep`, `wc`, `find`), Read, Grep, Write, one Explore subagent.

**Working directory:** Run directly on `main` (no worktree needed — read-only diagnostic, no risk of losing in-flight work). Verify at start: `git status` must be clean, `git worktree list` must show no active worktrees from a parallel session.

**Spec:** [docs/superpowers/specs/2026-04-20-ds-audit-design.md](../specs/2026-04-20-ds-audit-design.md)

## Task 2: Pass 2a — Read and classify top 20 worst pages

**Files:**
- Read: top 20 files from `docs/ds-audit/pass1-raw-counts.md` Section A
- Create: `docs/ds-audit/pass2-classifications.md`

**Purpose:** For each inline-style violation in the worst pages, classify it into AdoptAsIs / ExtendDS / NewDS. This is judgement-heavy and must stay in the main session.

- [ ] **Step 1: Create the classifications file with header**

Write `docs/ds-audit/pass2-classifications.md`:

```markdown
# Pass 2 — Violation classifications

For each file in the top-20-worst-pages list from Pass 1 Section A, every
flagged violation is classified:

- **AdoptAsIs** — existing DS component fits; page just chose to inline
- **ExtendDS:<Component>:<propName>** — DS component exists but needs a new prop
- **NewDS:<ProposedName>** — no existing component fits; new one needed

## Format per file

### <file path> — <violation count> violations

| # | Line | Snippet (truncated) | Violation category | Classification |
|---|------|---------------------|--------------------|----------------|
```

- [ ] **Step 2: For each of the top 20 pages, read the file and extract violations**

For each file:
1. Read the file with the Read tool
2. For every `style={{` occurrence, decide:
   - Is it pure layout (margin/padding/gap/flex/position/width/height/overflow/whiteSpace only)? Skip — not a violation per the spec's classifier.
   - Otherwise: which of `color`/`background`/`border`/`fontSize`/`fontWeight`/`boxShadow` properties are present? That's the violation category.
3. For each violation, decide classification:
   - Is there an existing DS component (Badge, Text, Button, Card, Divider, etc.) that already does this? → `AdoptAsIs`
   - Does an existing DS component almost do it but lacks a prop (e.g. a size, a tone, a shape, an onRemove)? → `ExtendDS:<Component>:<propName>`
   - None of the above — the pattern is unique or composed of multiple primitives? → `NewDS:<ProposedName>` (pick a descriptive name)
4. Add a row to the markdown table under that file's heading.

Iterate through all 20 files. For each classification, include enough snippet to understand the pattern (30-60 chars of code, truncate with `…`).
